## 📚 LLM Wiki — Schema

O vault Obsidian em `N8N/` é uma wiki persistente mantida exclusivamente pelo LLM. O humano lê; o LLM escreve e mantém.

### Camadas

| Camada | Localização | Dono | Regra |
|--------|-------------|------|-------|
| Fontes brutas | `workflows/` | Humano/n8nac | LLM **só lê**, nunca modifica |
| Wiki | `N8N/` | LLM | LLM cria, atualiza, mantém referências cruzadas |
| Schema | `AGENTS.md` (esta seção) | Co-evoluído | Define estrutura e convenções da wiki |

### Estrutura do Vault

```
N8N/
├── index.md              # Catálogo de todas as páginas (atualizar a cada ingestão)
├── log.md                # Registro cronológico append-only
├── overview.md           # Visão geral do ecossistema n8n
├── workflows/            # Uma página por workflow
├── conceitos/            # Páginas conceituais (tecnologias, padrões de negócio)
├── integracoes/          # Uma página por serviço externo integrado
└── padroes/              # Padrões arquiteturais reutilizáveis
```

### Frontmatter YAML (obrigatório em toda página)

```yaml
---
type: workflow | conceito | integracao | padrao | overview
tags: [tag1, tag2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources:
  - "[[Nome da Página Fonte]]"
---
```

### Operações

**Ingest** — Quando um novo workflow chega em `workflows/`:
1. Ler a fonte bruta (`.workflow.ts`)
2. Criar/atualizar a página do workflow em `N8N/workflows/`
3. Criar/atualizar páginas de conceitos, integrações e padrões relacionados
4. Atualizar `index.md` com as novas páginas
5. Adicionar entrada em `log.md`
6. Garantir que todos os wikilinks resolvam

**Query** — Quando o humano pergunta sobre a wiki:
1. Ler `index.md` primeiro para localizar páginas relevantes
2. Ler as páginas encontradas
3. Sintetizar resposta com citações `[[wikilink]]`
4. Se a resposta gerar análise valiosa, arquivá-la como nova página na wiki

**Lint** — Verificação periódica de integridade:
1. Procurar contradições entre páginas
2. Identificar páginas órfãs (sem links de entrada)
3. Identificar conceitos mencionados sem página própria
4. Verificar frontmatter YAML em todas as páginas
5. Verificar se `index.md` está completo

### Convenções

- **Idioma:** Português
- **Wikilinks:** Usar `[[Nome da Página]]` para links internos
- **Nomes de arquivo:** Descritivos, em português, com espaços
- **Mermaid:** Usar diagramas mermaid para fluxos visuais
- **Segurança:** Nunca expor tokens, API keys ou senhas na wiki. Referenciar como `(credencial gerenciada)` ou `(variável de ambiente)`
- **Formato do log:**

```markdown
## [YYYY-MM-DD] operação | Título
- **Fonte:** caminho do arquivo fonte
- **Páginas criadas:** N (detalhamento)
- **Páginas atualizadas:** lista
```

## 🎨 Regra de Frontend — UI/UX Pro Max (Obrigatória)

**Antes de qualquer tarefa relacionada a frontend**, o agente **DEVE** carregar a skill UI/UX Pro Max lendo o arquivo:

```
.agent/skills/ui-ux-pro-max/SKILL.md
```

### Quando ativar

Esta regra se aplica a **qualquer solicitação** que envolva:

- Criação, edição ou redesign de páginas web, componentes ou layouts
- Trabalho com HTML, CSS, JavaScript/TypeScript no contexto de interface
- Criação de dashboards, painéis, landing pages, formulários
- Escolha de cores, tipografia, espaçamentos, animações
- Frameworks frontend (React, Next.js, Vue, etc.)
- Geração de mockups ou protótipos visuais

### Como ativar

1. Ler o arquivo `SKILL.md` com a ferramenta `view_file` (com `IsSkillFile: true`)
2. Seguir todas as instruções, paletas, tipografias e padrões definidos pela skill
3. Aplicar as diretrizes de design **antes** de escrever qualquer código de interface

### Exceções

- Tarefas puramente de backend (APIs, banco de dados, n8n workflows) **não** precisam ativar esta skill
- Correções triviais de bugs que não alteram o visual (ex: fix de CORS, ajuste de rota) também são exceção
